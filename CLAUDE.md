# e2e-ai

AI-powered CLI tool that converts manual browser recordings into stable, documented Playwright tests. Published as `e2e-ai` on npm.

## Project Overview

This is a standalone npm package (not a monorepo). It provides a CLI (`e2e-ai`) and a library entry point.

### What it does

Pipeline: `record -> transcribe -> scenario -> generate -> refine -> test -> heal -> qa`

Each step uses an LLM agent (defined in `agents/*.md`) to transform artifacts. The package is framework-agnostic — project-specific knowledge comes from a user-generated `e2e-ai.context.md` file (created via `e2e-ai init`).

## Architecture

```
src/
  cli.ts                    # Entry point (commander.js) — registers all commands
  index.ts                  # Library entry: exports defineConfig, loadConfig, types
  config/
    schema.ts               # Zod v4 config schema + defineConfig()
    loader.ts               # Finds & loads e2e-ai.config.ts from user projects
    defaults.ts             # Full default config object
    paths.ts                # resolvePaths(config, key?) -> all derived paths
    playwright.ts           # Generates PW config at runtime from user config
  commands/
    _shared.ts              # resolveCommandContext() — shared CLI option resolution
    init.ts                 # Interactive setup wizard (scans codebase + LLM conversation)
    record.ts, transcribe.ts, scenario.ts, generate.ts,
    refine.ts, test.ts, heal.ts, qa.ts, run.ts
  pipeline/
    runPipeline.ts          # Sequential step runner with --from/--skip support
    types.ts                # PipelineContext, PipelineStep interfaces
  integrations/
    index.ts                # Dispatcher: fetchIssueContext(), generateQaExport()
    jira.ts, linear.ts      # Issue tracker adapters
    zephyr.ts               # Zephyr XML export
  agents/
    loadAgent.ts            # Two-part prompt: generic .md + project context file
    callLLM.ts              # Fetch-based OpenAI/Anthropic client (no SDK dep)
    types.ts                # AgentPrompt, LLMRequest, LLMResponse
  utils/
    fs.ts, logger.ts, process.ts

agents/                     # Agent prompt definitions (.md with YAML frontmatter)
scripts/                    # Bundled runtime scripts (codegen, voice, trace, auth)
templates/                  # Example files shipped with the package
```

## Key Design Decisions

- **Two-part prompt system**: Agent `.md` files contain generic instructions. Project-specific knowledge lives in `e2e-ai.context.md` (generated by `init`). These are composed at runtime in `loadAgent.ts`.
- **Config-driven paths**: All file paths are derived from `e2e-ai.config.ts` via `resolvePaths()`. No hardcoded project paths.
- **Integration abstraction**: `inputSource` (jira/linear/none) and `outputTarget` (zephyr/markdown/both) are config-driven. The dispatcher in `integrations/index.ts` routes accordingly.
- **Scripts use env vars**: `E2E_AI_PROJECT_ROOT`, `E2E_AI_WORKING_DIR`, `E2E_AI_KEY` — set by CLI commands, consumed by scripts.
- **No SDK dependencies for LLM**: `callLLM.ts` uses raw `fetch()` against OpenAI/Anthropic REST APIs.

```bash

bun install

# Build (ESM, split bundles)
bun run build
```

## Important Patterns

### Config Schema (Zod v4)
Uses `z.preprocess` for nested defaults because Zod v4's `.default({})` doesn't recursively apply inner field defaults:
```typescript
function nested<T extends z.ZodRawShape>(shape: T) {
  const inner = z.object(shape);
  return z.preprocess((val) => val ?? {}, inner);
}
```

### Agent Loading
```typescript
// loadAgent('scenario-agent', config) does:
// 1. Read agents/scenario-agent.md (generic prompt)
// 2. Read e2e-ai.context.md from user's project (if exists)
// 3. Compose: generic + "\n\n## Project Context\n\n" + context
// 4. Apply per-agent model overrides from config.llm.agentModels
```

### Path Resolution
```typescript
// getProjectRoot() — walks up from cwd to find e2e-ai.config.ts
// getPackageRoot() — resolves from import.meta.dirname (works in node_modules too)
// resolvePaths(config, key?) — returns all derived paths from config
```

## Conventions

- All source in TypeScript ESM (`"type": "module"`)
- Use `.ts` extensions in imports (Bun resolves them)
- Commands register via `register<Name>(program: Command)` pattern
- Agent prompts use YAML frontmatter: `model`, `max_tokens`, `temperature`
- No project-specific references in agents or source — use generic placeholders (PROJ-101, LIN-42)
- `scripts/auth/setup-auth.mjs` is a TEMPLATE — users adapt it to their login flow
